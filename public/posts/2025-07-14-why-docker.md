# Docker를 왜 사용할까?

"내 컴퓨터에서는 잘 되는데요." 협업을 하다 보면 한 번쯤 마주치는 상황입니다. 같은 코드인데 팀원 환경에서는 실행이 안 됩니다. Node.js 버전이 다르거나, 로컬에만 설치된 라이브러리가 있거나, OS 차이 때문입니다.

Docker는 이 문제를 근본적으로 해결합니다. 코드뿐 아니라 **실행 환경 자체를 패키징**해서 어디서든 동일하게 실행할 수 있게 만들어주는 기술입니다.

---

## 환경 차이가 만드는 문제

프로그램이 정상적으로 실행되려면 코드만 있어서는 안 됩니다. 런타임 버전, 설치된 패키지, 환경 변수, OS 등이 모두 맞아야 합니다. 이걸 통틀어 **실행 환경**이라고 합니다.

개발자마다 환경이 조금씩 다릅니다. 이런 차이를 줄이기 위해 예전에는 **가상 머신(VM)** 을 사용했지만, VM은 OS 전체를 가상화하기 때문에 무겁고 시작이 느립니다.

---

## Docker의 핵심 개념

Docker는 가상 머신과 달리 OS를 통째로 가상화하지 않습니다. 호스트 OS의 커널을 공유하면서 프로세스만 격리하는 방식으로, 훨씬 가볍고 빠릅니다.

**이미지(Image)**
실행 환경의 스냅샷입니다. 어떤 OS 위에서 어떤 런타임을 쓰고, 어떤 패키지를 설치하는지를 `Dockerfile`로 정의하면 이미지가 만들어집니다.

**컨테이너(Container)**
이미지를 실행한 인스턴스입니다. 이미지 하나로 여러 컨테이너를 실행할 수 있고, 각 컨테이너는 서로 격리됩니다.

```dockerfile
# Dockerfile 예시
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
CMD ["node", "dist/main.js"]
```

이 파일 하나로 팀 전체가 동일한 환경을 공유합니다. `docker build`로 이미지를 만들고, `docker run`으로 실행하면 끝입니다.

---

## Docker를 쓰는 이유

**환경 일치**
로컬 개발 환경과 서버 환경을 동일한 이미지로 맞출 수 있습니다. "로컬에서는 잘 되는데 서버에서 안 된다"는 문제가 사라집니다.

**빠른 환경 구성**
새 팀원이 합류하거나 새 서버를 구성할 때, 복잡한 설치 과정 없이 `docker run` 한 줄로 환경을 재현할 수 있습니다.

**격리**
프로젝트마다 다른 Node.js 버전이나 라이브러리를 써야 할 때, 컨테이너로 분리해 충돌 없이 관리할 수 있습니다.

**가볍고 빠름**
VM과 달리 초 단위로 컨테이너를 생성하고 삭제할 수 있습니다. CI/CD 파이프라인에서 특히 유용합니다.

---

## VM과 Docker 비교

| | 가상 머신 (VM) | Docker |
|---|---|---|
| 가상화 대상 | OS 전체 | 프로세스만 격리 |
| 시작 시간 | 수십 초 ~ 수 분 | 수 초 이내 |
| 이미지 크기 | GB 단위 | MB 단위 |
| 성능 오버헤드 | 높음 | 낮음 |

---

## 언제 Docker가 특히 유용할까?

- **팀 협업**: 환경 차이로 인한 문제를 없앨 수 있습니다.
- **CI/CD**: GitHub Actions 같은 파이프라인에서 이미지를 빌드하고 배포하는 흐름을 표준화할 수 있습니다.
- **마이크로서비스**: 서비스별로 독립적인 컨테이너로 분리해 배포할 수 있습니다.
- **로컬 DB 환경**: PostgreSQL, Redis 등을 로컬에 직접 설치하지 않고 컨테이너로 실행할 수 있습니다.

Docker는 서버 배포 도구이기도 하지만, 협업과 개발 환경 표준화 도구로서도 가치가 높습니다.
