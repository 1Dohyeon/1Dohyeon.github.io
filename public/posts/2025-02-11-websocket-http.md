# WebSocket과 HTTP: 실시간 통신은 어떻게 작동할까?

채팅, 알림, 실시간 협업 같은 기능을 구현하려고 하면 HTTP만으로는 한계에 부딪힙니다. 클라이언트가 요청해야만 서버가 응답하는 HTTP 구조에서는 "서버가 먼저 데이터를 보내는" 것이 기본적으로 불가능하기 때문입니다.

WebSocket은 이 문제를 해결하기 위해 등장한 프로토콜입니다. HTTP와 WebSocket이 어떻게 다른지, 그리고 WebSocket 이전에 실시간 통신을 어떻게 흉내냈는지부터 살펴봅니다.

---

## HTTP의 구조적 한계

HTTP는 **클라이언트가 요청하면 서버가 응답하는** 반이중(Half-Duplex) 통신입니다. 요청-응답이 한 쌍으로 묶여 있고, 응답이 끝나면 연결이 닫힙니다.

이 구조에서 실시간 기능을 구현하려면 다음과 같은 우회 방법을 써야 합니다.

**Polling(폴링)**
클라이언트가 일정 주기로 서버에 계속 요청을 보냅니다. 새 데이터가 없어도 요청이 발생하므로 불필요한 트래픽이 많습니다.

**Long Polling(롱 폴링)**
클라이언트가 요청을 보내면 서버가 응답을 보류하다가 데이터가 생기면 그때 응답합니다. Polling보다는 효율적이지만 연결을 오래 물고 있다는 한계가 있습니다.

**Server-Sent Events(SSE)**
클라이언트가 한 번 요청하면 서버가 연결을 끊지 않고 데이터가 생길 때마다 계속 전송합니다. 서버 → 클라이언트 단방향 스트리밍에는 적합하지만, 클라이언트가 서버로 데이터를 보내려면 별도의 HTTP 요청이 필요합니다.

세 방식 모두 진정한 실시간 양방향 통신이라고 보기 어렵습니다.

---

## WebSocket: 연결을 유지하는 양방향 통신

WebSocket은 **전이중(Full-Duplex)** 통신을 지원합니다. 연결이 맺어진 뒤에는 클라이언트와 서버 모두 언제든지 상대방에게 데이터를 보낼 수 있습니다.

### Handshake: HTTP에서 WebSocket으로

WebSocket 연결은 HTTP 요청으로 시작합니다. 클라이언트가 서버에 "WebSocket으로 업그레이드하자"는 요청을 보내고, 서버가 이를 수락하면 이후부터는 WebSocket 프로토콜로 통신합니다.

```
// 클라이언트 → 서버 (HTTP Upgrade 요청)
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

// 서버 → 클라이언트 (업그레이드 수락)
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

이 Handshake가 완료되면 HTTP 연결이 WebSocket 연결로 전환되고, 이후 연결은 명시적으로 닫기 전까지 유지됩니다.

### TCP 기반의 신뢰성

WebSocket은 TCP 위에서 동작합니다. TCP는 패킷의 순서와 전달을 보장하기 때문에, 데이터가 손실되거나 순서가 뒤바뀌는 일이 없습니다. 채팅 메시지나 실시간 상태 동기화처럼 순서가 중요한 데이터를 안정적으로 주고받을 수 있습니다.

---

## HTTP vs WebSocket 비교

| | HTTP | WebSocket |
|---|---|---|
| 통신 방향 | 단방향 (요청-응답) | 양방향 |
| 연결 유지 | 응답 후 종료 | 지속 연결 |
| 서버 → 클라이언트 전송 | 불가 (우회 필요) | 가능 |
| 오버헤드 | 요청마다 헤더 포함 | 연결 후 최소화 |
| 적합한 상황 | 일반 API, 페이지 요청 | 채팅, 알림, 실시간 협업 |

---

## 어떤 상황에 WebSocket을 선택할까?

양방향 통신이 필요하거나, 서버가 먼저 데이터를 전송해야 하는 상황이라면 WebSocket이 적합합니다.

- 실시간 채팅
- 게임 상태 동기화
- 협업 도구 (공동 편집)
- 실시간 알림

반면 단순한 데이터 조회나 폼 제출처럼 요청-응답 패턴이 명확한 경우는 HTTP REST API가 더 적합합니다. WebSocket은 연결을 유지하는 비용이 있기 때문에, 필요하지 않은 곳에 쓰면 오히려 서버 자원을 낭비합니다.
