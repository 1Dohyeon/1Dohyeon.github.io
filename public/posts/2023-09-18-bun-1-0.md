# Bun 1.0: Node.js 생태계의 도전자

2023년 9월, Bun 1.0이 정식 출시됐습니다. 출시 전부터 "Node.js보다 훨씬 빠르다"는 벤치마크로 주목받았는데, 실제로 무엇이 다르고 어떤 상황에서 의미 있는지 살펴봅니다.

---

## Bun은 무엇인가

Bun은 JavaScript/TypeScript 런타임입니다. Node.js와 Deno가 V8(Chrome 엔진)을 사용하는 것과 달리, Bun은 **JavaScriptCore(Safari 엔진)** 를 사용합니다. 또한 성능에 민감한 부분은 **Zig 언어**로 작성됐습니다.

런타임 외에도 Bun은 하나의 도구 안에 여러 기능을 통합했습니다.

| 기능 | 기존 도구 | Bun |
|---|---|---|
| 런타임 | Node.js | Bun |
| 패키지 매니저 | npm, yarn, pnpm | `bun install` |
| 번들러 | webpack, esbuild, Rollup | `bun build` |
| 테스트 러너 | Jest, Vitest | `bun test` |
| TypeScript 실행 | ts-node, tsx | 기본 지원 |

---

## 실제로 얼마나 빠른가

[공식 벤치마크](https://bun.sh/blog/bun-v1.0) 기준으로 Bun의 속도는 인상적입니다.

- **패키지 설치**: npm 대비 최대 30배 빠름
- **서버 처리량**: Node.js 대비 약 4배 높은 HTTP 요청 처리
- **스크립트 시작 시간**: Node.js보다 빠른 cold start

물론 벤치마크는 조건에 따라 달라집니다. 공식 수치는 인상적이지만, 패키지 설치 속도 차이가 가장 직접적으로 느껴지는 부분인 것 같습니다. 서버 처리량 차이는 작업 특성에 따라 다르다고 합니다.

---

## Node.js 호환성

Bun이 실용적이려면 기존 Node.js 생태계를 그대로 쓸 수 있어야 합니다. 1.0 기준으로 Node.js API의 대부분을 지원하며, `node_modules`와 `package.json`을 그대로 사용합니다. 많은 Node.js 프레임워크(Express, Fastify, NestJS 등)가 Bun 위에서 실행됩니다.

다만 100% 호환은 아닙니다. 특히 Node.js의 내부 모듈 중 일부나 특정 네이티브 애드온은 동작하지 않을 수 있습니다.

```bash
# 기존 Node.js 프로젝트에서 그대로 사용 가능
bun install       # npm install 대체
bun run dev       # npm run dev 대체
bun index.ts      # TypeScript 직접 실행 (ts-node 불필요)
```

## Deno와 어떻게 다른가

Deno도 Bun과 비슷하게 Node.js의 대안으로 나왔지만 전략이 다릅니다.

Deno는 Node.js와의 **단절**을 선택했습니다. 보안 모델을 새로 설계하고, 기본 모듈 시스템을 URL 기반으로 바꿨습니다. 덕분에 설계는 깔끔하지만 기존 Node.js 생태계와 호환이 어렵습니다.

Bun은 반대로 **호환성을 최우선**으로 했습니다. Node.js 코드를 최대한 그대로 실행할 수 있게 하면서 속도만 높이는 전략입니다. 진입 장벽이 낮습니다.

## 지금 바로 마이그레이션해야 할까

아직은 신중하게 접근하는 것이 좋습니다.

**써볼 만한 상황**
- 개인 프로젝트나 사이드 프로젝트
- TypeScript를 ts-node 없이 바로 실행하고 싶을 때
- 패키지 설치 속도를 직접 비교해보고 싶을 때

**아직 조심스러운 상황**
- 안정성이 중요한 프로젝트
- 특정 네이티브 모듈에 의존하는 경우

Bun은 분명히 빠르고, 개발 경험도 좋습니다. 다만 생태계 성숙도는 Node.js와 아직 격차가 있어서, 개인 프로젝트에서 먼저 써보면서 감을 익히는 게 좋을 것 같습니다.
