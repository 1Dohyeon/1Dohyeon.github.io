# 데이터베이스 인덱스(Index)는 어떻게 작동할까?

데이터가 수천 건일 때는 아무 문제 없던 조회 쿼리가, 수백만 건이 쌓이면 갑자기 느려집니다. `SELECT` 한 줄에 수 초가 걸리기 시작하는 시점이 오죠. 이 문제를 해결하는 가장 기본적인 도구가 **인덱스(Index)** 입니다.

인덱스가 무엇인지, 내부적으로 어떻게 동작하는지, 그리고 언제 써야 하는지 정리합니다.

---

## 인덱스란?

인덱스는 책의 **색인(Index)** 과 같습니다. 책에서 특정 키워드를 찾을 때, 앞 페이지부터 한 장씩 넘기는 대신 색인에서 페이지 번호를 바로 찾아 이동합니다.

데이터베이스도 동일합니다. 인덱스가 없다면 `WHERE email = 'user@example.com'` 조건 하나를 처리하기 위해 테이블의 모든 행을 처음부터 끝까지 스캔해야 합니다. 이를 **Full Scan**이라고 합니다. 데이터가 100만 건이면 100만 번 비교합니다.

인덱스를 만들면 해당 컬럼의 값과 행 위치 정보를 별도의 자료구조에 저장해두고, 조회 시 그 자료구조를 먼저 탐색해 위치를 특정한 뒤 바로 접근합니다.

---

## 내부 구조: B-Tree

데이터베이스 인덱스는 대부분 **B-Tree(Balanced Tree)** 구조로 구현됩니다.

B-Tree는 항상 정렬된 상태를 유지하며, 어떤 값을 찾더라도 루트에서 리프 노드까지의 깊이가 일정합니다. 덕분에 탐색 시간이 `O(log N)`으로 보장됩니다.

예를 들어 `WHERE name = '홍길동'`을 처리할 때:

- 인덱스 없음 → 전체 행을 순차 스캔 (`O(N)`)
- 인덱스 있음 → B-Tree에서 '홍길동' 키를 탐색한 뒤 해당 행의 위치(Row ID)로 바로 접근 (`O(log N)`)

100만 건 기준으로 Full Scan은 최대 100만 번 비교지만, B-Tree 탐색은 약 20번 이내로 끝납니다.

---

## 트레이드오프: 쓰기 성능

인덱스는 조회 속도를 높이는 대신 **쓰기 성능을 낮춥니다.**

데이터를 삽입, 수정, 삭제할 때마다 인덱스도 함께 갱신해야 하기 때문입니다. 인덱스가 많을수록 쓰기 작업 비용이 늘어납니다. 또한 인덱스 자체도 디스크 공간을 차지합니다.

| | 인덱스 없음 | 인덱스 있음 |
|---|---|---|
| 조회 속도 | 느림 (Full Scan) | 빠름 (B-Tree 탐색) |
| 쓰기 속도 | 빠름 | 느림 (인덱스 갱신) |
| 저장 공간 | 적음 | 많음 |

---

## 언제 인덱스를 써야 할까?

**적합한 경우**
- `WHERE`, `ORDER BY`, `GROUP BY`, `JOIN` 조건에 자주 등장하는 컬럼
- 데이터 양이 많고 조회가 잦은 테이블
- 자주 읽히지만 수정은 적은 컬럼 (`email`, `username` 등)

**피해야 할 경우**
- 데이터 양이 적은 테이블 (Full Scan이 오히려 빠를 수 있음)
- 쓰기가 매우 잦은 컬럼
- 카디널리티(고유값 비율)가 낮은 컬럼 (예: `boolean` 타입 컬럼)

인덱스를 무분별하게 늘리면 쓰기 성능이 전반적으로 저하됩니다. **자주 조회되는 컬럼에만 선택적으로** 적용하는 것이 원칙입니다.
