# JWT vs 세션, 어떤 로그인 방식을 선택할까?

로그인 기능을 구현하고 나면 다음 고민이 시작됩니다. "로그인 상태를 어떻게 유지할까?" 가장 많이 쓰이는 두 가지 방식은 **서버 세션**과 **JWT 토큰**입니다. 두 방식의 개념과 트레이드오프를 정리하고, JWT + HttpOnly 쿠키를 선택한 이유를 설명합니다.

---

## 서버 세션 방식

사용자가 로그인하면 서버가 해당 사용자의 정보를 메모리나 DB에 저장하고, 클라이언트 브라우저에는 **세션 ID**가 담긴 쿠키를 전달합니다. 이후 요청마다 세션 ID를 함께 보내면 서버가 ID로 사용자를 식별합니다.

**장점**
- 서버가 인증 정보를 직접 관리해 구현이 단순
- 세션을 서버에서 즉시 무효화할 수 있음 (강제 로그아웃 가능)

**단점**
- 서버가 상태를 저장해야 하므로, 서버가 여러 대일 경우 세션을 공유하는 별도 구성이 필요 (Redis, Sticky Session 등)
- 수평 확장이 어려움

---

## JWT 방식

사용자가 로그인하면 서버가 인증 정보를 담은 **JWT(JSON Web Token)** 를 생성해 클라이언트에 전달합니다. 이후 클라이언트가 요청마다 JWT를 함께 보내면, 서버는 서명을 검증해 사용자를 식별합니다.

JWT는 **Header.Payload.Signature** 세 부분으로 구성됩니다. Payload에는 사용자 정보와 만료 시간이 담기고, Signature는 secret key로 서명해 위변조를 방지합니다.

**장점**
- 서버가 상태를 저장하지 않아 수평 확장에 유리
- 마이크로서비스 환경에 적합

**단점**
- 토큰이 클라이언트에 저장되므로 탈취 시 대응이 어려움
- 만료 전까지 서버에서 강제 무효화가 어려움

---

## JWT + HttpOnly 쿠키를 선택한 이유

기본적인 보안을 고려해 **JWT 토큰을 HttpOnly 쿠키에 담는 방식**을 선택했습니다. 두 가지 공격을 방어하기 위해서입니다.

- **XSS 방어**: `HttpOnly` 쿠키는 JavaScript에서 접근이 불가능해, 악성 스크립트로 토큰을 탈취할 수 없습니다.
- **중간자 공격 방어**: `Secure` 속성으로 HTTPS에서만 전송되도록 제한합니다.

```ts
res.cookie("accessToken", accessToken, {
    httpOnly: true,   // JavaScript 접근 차단 (XSS 방어)
    secure: true,     // HTTPS에서만 전송
    sameSite: "strict", // 외부 사이트 요청에 쿠키 미포함 (CSRF 방어)
    maxAge: 15 * 60 * 1000, // 15분 만료
});
```

`accessToken`은 15분으로 짧게 설정하고, 프론트엔드에서 10분마다 `/auth/refresh`를 호출해 자동 갱신합니다. 브라우저가 쿠키를 자동으로 포함시키므로 사용자는 별도로 토큰을 관리할 필요가 없습니다.

```ts
setInterval(async () => {
    const res = await api.auth.refresh();
    if (!res.success) {
        await logout();
    }
}, 10 * 60 * 1000);
```

## JWT의 한계: 강제 무효화

JWT는 서버가 상태를 저장하지 않는 구조라, 발급한 토큰을 중간에 무효화하기 어렵습니다. 로그아웃을 하거나 계정이 탈취되어도, 토큰 자체는 만료 시간 전까지 유효합니다.

이를 해결하기 위해 **토큰 블랙리스트** 방식을 고려했습니다. 무효화할 토큰을 Redis에 등록하고, 이후 요청마다 블랙리스트를 조회해 해당 토큰이 있으면 거부하는 방식입니다.

```ts
// 로그아웃 시 블랙리스트 등록 예시
await redis.set(`blacklist:${token}`, "1", "EX", remainingTtl);

// 인증 미들웨어에서 검증
const isBlacklisted = await redis.get(`blacklist:${token}`);
if (isBlacklisted) throw new UnauthorizedException();
```

단, 이 방식은 JWT의 핵심 장점인 **무상태성(Stateless)** 을 포기하는 트레이드오프가 있습니다. 무상태성을 지키려면 `accessToken` 만료 시간을 충분히 짧게 설정하는 것으로 리스크를 줄이는 방향을 선택할 수 있습니다.

---

## 정리

| | 세션 | JWT |
|---|---|---|
| 상태 저장 | 서버 | 클라이언트 |
| 확장성 | 낮음 (세션 공유 필요) | 높음 |
| 강제 무효화 | 즉시 가능 | 어려움 (블랙리스트 필요) |
| 적합한 상황 | 단일 서버, 즉시 로그아웃 필요 | 분산 환경, 마이크로서비스 |

두 방식 모두 장단점이 있습니다. 서비스의 규모와 보안 요구사항에 따라 선택이 달라집니다.
