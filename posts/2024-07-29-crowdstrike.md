# CrowdStrike 사태: 소프트웨어 업데이트는 왜 위험한가

2024년 7월 19일, 전 세계 850만 대 이상의 Windows 시스템이 블루스크린(BSOD)으로 다운됐습니다. 공항, 병원, 방송사까지 영향을 받은 대규모 장애였습니다. [CrowdStrike 공식 분석](https://www.crowdstrike.com/blog/falcon-update-for-windows-hosts-technical-details/)을 읽어봤습니다.

---

## 무슨 일이었나

보안 솔루션 회사 CrowdStrike가 Falcon 센서의 설정 파일(Channel File 291)을 업데이트했는데, 이 파일에 논리 오류가 있었습니다. 보안 소프트웨어는 커널 레벨에서 동작하기 때문에 오류가 OS 전체를 다운시켰습니다.

```
Channel File 291 업데이트 배포
→ Falcon 센서가 파일 로드 시 잘못된 메모리 접근
→ 커널 패닉
→ BSOD (블루스크린)
```

---

## 왜 이렇게 빠르게 퍼졌나

CrowdStrike는 실시간 위협 대응을 위해 설정 파일을 자동으로 배포합니다. 이 자동 배포 덕분에 수백만 대에 동시 배포됐고, 문제도 동시에 퍼졌습니다.

보안 소프트웨어의 특성상 자동 업데이트가 핵심 기능인데, 그 핵심 기능이 동시에 취약점이 된 상황입니다.

---

## 소프트웨어 배포에서 배운 것

**카나리 배포(Canary Deployment)**
전체 시스템에 한 번에 배포하지 않고, 일부에 먼저 배포해서 이상이 없으면 점진적으로 확대하는 방식입니다. CrowdStrike가 이 방식을 사용했다면 피해 규모가 훨씬 작았을 것입니다.

**커널 레벨 소프트웨어의 위험성**
일반 애플리케이션은 오류가 나도 그 앱만 종료됩니다. 커널 레벨에서 동작하는 소프트웨어는 오류가 시스템 전체에 영향을 줍니다. 그만큼 더 엄격한 테스트와 배포 전략이 필요합니다.

개발 공부를 하면서 배포 전략이 왜 중요한지 막연하게만 알고 있었는데, 이 사태를 보면서 실감이 됐습니다. 빠른 배포가 항상 좋은 게 아니라는 것, 점진적 배포의 중요성이 이렇게 극명하게 드러난 사례는 처음 봤습니다.
