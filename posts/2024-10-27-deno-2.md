# Deno 2.0: Node.js와의 호환성을 선택한 이유

[Deno 2.0](https://deno.com/blog/v2)이 출시됐습니다. 가장 눈에 띄는 변화는 Node.js·npm과의 호환성을 대폭 강화했다는 점입니다. Deno 1.x가 추구하던 방향과는 꽤 다릅니다.

---

## Deno 1.x는 왜 쓰기 어려웠나

Deno는 Node.js의 설계 실수를 고치겠다는 목표로 만들어졌습니다. Ryan Dahl(Node.js 창시자)이 직접 만든 프로젝트입니다.

- `node_modules` 없이 URL 기반 모듈 시스템
- 기본 보안 샌드박스 (파일·네트워크 접근 시 명시적 허용 필요)
- 내장 TypeScript 지원

설계는 깔끔했지만 npm 생태계와 호환이 안 돼서 기존 패키지를 그대로 쓸 수 없었습니다. 실제로 쓰기가 어려웠습니다.

---

## 2.0에서 바뀐 것

npm 패키지를 `npm:` 접두사로 직접 사용할 수 있고, `node_modules`도 지원합니다.

```typescript
// Deno 2.0에서 npm 패키지 사용
import express from "npm:express";
import { z } from "npm:zod";
```

기존 Node.js API 호환성도 크게 높아졌습니다. 많은 Node.js 프레임워크가 별도 수정 없이 Deno 위에서 실행됩니다.

---

## 왜 방향을 바꿨을까

호환성을 포기하고 더 나은 설계를 추구했더니 사용자가 늘지 않더라는 현실적인 판단인 것 같습니다. 보안 샌드박스, 내장 TypeScript 지원, `deno fmt`·`deno lint` 같은 내장 도구들은 그대로 유지하면서 진입 장벽만 낮췄습니다.

설계의 순수함보다 생태계 호환성이 채택에 훨씬 중요하다는 걸 보여주는 사례입니다. Node.js가 여전히 npm 생태계를 등에 업고 있는 한, 호환성을 포기하는 건 현실적으로 어렵다는 것을 Deno가 몸소 보여준 것 같습니다.
