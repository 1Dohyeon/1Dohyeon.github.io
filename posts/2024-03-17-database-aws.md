# Database와 AWS 기초 개념 정리

서버를 개발하기 시작하면 MySQL, MongoDB, AWS 같은 용어를 금방 마주치게 됩니다. 각각이 무엇인지, 어떤 맥락에서 쓰이는지 이해하지 못한 채 따라가다 보면 나중에 설계 결정을 내릴 때 근거가 없어집니다. 데이터베이스의 종류와 특성, 그리고 클라우드 인프라의 기본 개념을 순서대로 정리합니다.

---

## Database란?

데이터베이스는 **구조화된 데이터의 모음**입니다. 애플리케이션이 생성하는 유저 정보, 게시글, 주문 내역 같은 데이터를 저장하고 조회할 수 있도록 체계적으로 관리합니다.

데이터베이스 자체는 파일의 집합에 가깝고, 이를 실제로 조작할 수 있게 해주는 소프트웨어가 **DBMS(Database Management System)** 입니다. 우리가 흔히 "MySQL을 쓴다", "MongoDB를 쓴다"고 말할 때, 그 대상이 바로 DBMS입니다.

---

## DBMS의 종류

DBMS는 데이터를 저장하는 방식에 따라 여러 유형으로 나뉩니다.

- 관계형 데이터베이스 (RDBMS) - MySQL, PostgreSQL, Oracle
- 문서형 (Document) - MongoDB
- 키-값 (Key-Value) - Redis
- 그래프 (Graph) - Neo4j
- 와이드 컬럼 (Wide Column) - Cassandra

가장 많이 쓰이는 두 축은 **RDBMS**와 **NoSQL**입니다. 나머지 유형은 특수한 목적에 맞게 선택합니다.

---

## SQL이란?

SQL(Structured Query Language)은 관계형 데이터베이스를 조작하기 위한 언어입니다. 특정 회사의 제품이 아니라 국제표준화기구(ISO)가 표준을 정의하고, 각 DBMS 벤더가 이를 기반으로 자체 확장 문법을 추가합니다.

```sql
SELECT * FROM users WHERE email = 'example@email.com';
```

MySQL, PostgreSQL, Oracle 모두 SQL을 사용하지만, 세부 문법에 차이가 있습니다. 표준 SQL을 익혀두면 DBMS가 바뀌어도 빠르게 적응할 수 있습니다.

---

## RDBMS: 테이블 기반의 관계형 데이터베이스

RDBMS는 데이터를 **테이블(표)** 형태로 저장합니다. 행(row)은 하나의 데이터 레코드, 열(column)은 데이터의 속성을 나타냅니다.

| id  | name    | email        |
| --- | ------- | ------------ |
| 1   | Example | ex@email.com |
| 2   | Gildong | gd@email.com |

각 테이블에는 **Primary Key**라는 고유 식별자가 있고, 다른 테이블의 **Foreign Key**로 연결해 관계를 표현합니다. "관계형"이라는 이름이 여기서 나옵니다.

**장점**

- 데이터 무결성 보장 - 중복, 누락 없이 정확한 데이터 유지
- 스키마가 명확히 정의되어 있어 데이터 일관성이 높음
- 트랜잭션 지원으로 안전한 데이터 처리 가능

**단점**

- 스키마를 사전에 정의해야 해서 구조 변경이 어려움
- 수평 확장(서버를 늘리는 방식)보다 수직 확장(서버 사양을 올리는 방식)에 적합

---

## NoSQL: 유연한 비관계형 데이터베이스

NoSQL은 "Not Only SQL"의 약자로, 관계형 모델을 따르지 않는 데이터베이스를 통칭합니다. MongoDB는 그 중 **문서형(Document)** 방식으로, 데이터를 JSON 형태의 Document로 저장합니다.

```json
{
  "_id": "2JID92JIA2FOSPF",
  "name": "Dohyeon",
  "email": "dh@email.com",
  "posts": [{ "title": "첫 번째 글", "createdAt": "2024-03-17" }]
}
```

관련 데이터를 하나의 Document 안에 중첩해서 담을 수 있어, 조인 없이 한 번의 조회로 필요한 데이터를 가져올 수 있습니다.

**장점**

- 스키마가 유연해 구조 변경이 쉬움
- 수평 확장에 유리 - 서버를 추가하는 방식으로 확장 가능
- 대량의 데이터와 높은 트래픽에 적합

**단점**

- 데이터 일관성 보장이 RDBMS보다 약함
- 복잡한 관계 표현에는 적합하지 않음

Node.js 환경에서는 MongoDB와 함께 **Mongoose**를 자주 사용합니다. Mongoose는 MongoDB를 Node.js에서 다루기 위한 ODM(Object Data Modeling) 라이브러리로, 스키마 정의와 유효성 검사를 편리하게 처리할 수 있게 해줍니다.

---

## RDBMS vs NoSQL: 언제 무엇을 선택할까?

두 방식은 우열이 아니라 **적합한 상황이 다릅니다.**

|             | RDBMS                 | NoSQL                   |
| ----------- | --------------------- | ----------------------- |
| 데이터 구조 | 고정 스키마           | 유연한 스키마           |
| 확장 방식   | 수직 확장             | 수평 확장               |
| 일관성      | 강함                  | 상대적으로 약함         |
| 적합한 상황 | 금융, 결제, 유저 관리 | 로그, 피드, 대용량 처리 |

금융처럼 데이터 정확성이 중요한 도메인은 RDBMS, 소셜 피드나 로그처럼 스키마가 자주 바뀌고 대용량인 경우는 NoSQL이 적합합니다. 실제 서비스에서는 두 가지를 함께 사용하는 경우도 많습니다.

---

## AWS와 클라우드 컴퓨팅

데이터베이스를 로컬 환경에서만 운영하면 한계가 분명합니다. 서버가 꺼지면 서비스도 꺼지고, 용량이 부족하면 사양을 직접 올려야 합니다.

**클라우드 컴퓨팅**은 인터넷을 통해 컴퓨팅 자원(서버, 스토리지, 데이터베이스 등)을 필요한 만큼 빌려 쓰는 방식입니다. 물리적인 서버를 직접 구매하거나 관리할 필요 없이, 사용한 만큼만 비용을 냅니다.

**AWS(Amazon Web Services)** 는 이 클라우드 서비스 시장의 최대 사업자입니다. 주요 서비스만 정리하면 다음과 같습니다.

| 서비스     | 역할                       |
| ---------- | -------------------------- |
| EC2        | 가상 서버 (컴퓨팅)         |
| RDS        | 관리형 관계형 데이터베이스 |
| S3         | 파일 스토리지              |
| Lambda     | 서버리스 함수 실행         |
| CloudFront | CDN (콘텐츠 전송 네트워크) |

직접 서버를 구축하는 온프레미스(On-premise) 방식과 비교했을 때, 클라우드의 가장 큰 장점은 **확장성**입니다. 트래픽이 갑자기 늘어나도 몇 분 안에 서버를 추가할 수 있고, 줄어들면 다시 줄일 수 있습니다. 초기 인프라 비용 없이 서비스를 시작할 수 있다는 점도 스타트업과 개인 개발자에게 유리합니다.
