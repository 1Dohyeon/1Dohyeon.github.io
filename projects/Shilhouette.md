# Shilhouette: 일정 관리 및 소셜 미디어

## 프로젝트 개요

- **개발 기간**: 2025.08 ~ 2025.09 (약 3주)
- **기술 스택**: Next.js, NestJS, PostgreSQL, GCP Cloud Storage
- **협업 인원**: 3인
- **맡은 역할**: 프론트엔드 전체, 사용자 인증 · 일정 관리 API · 미디어 파일 업로드 백엔드, GCP 스토리지 아키텍처 설계

> [UI Repository](https://github.com/DOforTU/talkflow-web) · [Server Repository](https://github.com/DOforTU/talkflow-server)

## 왜 이 프로젝트를 시작했는가

일정 관리 앱을 쓰면서 할 일을 체크하는 것만으로는 성취감이 부족하다고 느꼈습니다. 더 나은 TODO 관리 서비스를 만들어보자는 것이 출발점이었습니다.

개발 중 팀원들과 논의하며 방향이 바뀌었습니다. 기존 소셜 미디어는 좋아요 수, 팔로워 수 같은 숫자에 집착하게 만들고 끝없는 스크롤로 시간을 소모하게 합니다. 이 피로감에 대한 대안으로 **'조용한 소셜 미디어'** 컨셉을 추가했습니다.

구체적으로는 세 가지 제약을 의도적으로 설계했습니다. 하루 최대 2번만 공유할 수 있고, 콘텐츠는 사진 또는 15초 이내 영상으로 제한됩니다. 본인 게시물에도 공감 수를 볼 수 없고, 여러 명이 공감했다는 알림만 받습니다. 숫자 경쟁 없이 일상을 기록하고 소통하는 데 집중하는 구조입니다.

## 왜 중단했는가

MVP를 완성한 뒤 프로젝트를 중단했습니다. '조용한 소셜 미디어'라는 컨셉이 시장에서 검증되지 않은 상태에서 일정 관리와 소셜 기능 두 가지를 동시에 잡으려다 보니 핵심 가치가 흔들렸습니다. 사용자에게 전달하려는 것이 '더 나은 일정 관리'인지 '다른 방식의 소통'인지 팀 안에서도 명확히 정의하지 못했습니다.

3주라는 시간 안에 두 가지 방향을 모두 완성도 있게 구현하기도 어려웠고, 결국 집중할 우선순위를 정하지 못한 채 마무리했습니다.

아이디어 자체에 대한 확신은 여전히 있습니다. 충분한 시간과 명확한 방향 설정이 가능하다면 다시 검증해보고 싶은 프로젝트입니다. 기능의 범위를 제한하는 것이 서비스의 핵심 가치를 더 선명하게 만든다는 것, 그리고 새로운 컨셉은 구현보다 시장 검증이 먼저라는 것을 이 경험에서 배웠습니다.
