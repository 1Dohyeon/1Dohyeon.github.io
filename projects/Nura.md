# Nura

![]()

## 개발 동기

[NewLearn Note](https://1dohyeon.github.io/projects/newlearnnote)를 기획하면서 핵심 기능 중 하나로 AI 문서 분석을 염두에 두고 있었습니다. 그런데 RAG 파이프라인을 처음 다뤄보는 상황에서 메인 프로젝트에 바로 통합하기가 꺼려졌습니다. 문서 파싱, 청킹, 임베딩, 벡터 검색, 응답 생성까지 각 단계에서 어떤 문제가 생길지 예측하기 어려웠고, 청킹 전략이나 프롬프트 설계를 실험해볼 독립적인 환경이 필요했습니다.

그래서 Nura를 별도 프로토타입으로 개발해 RAG 파이프라인을 먼저 검증하고, 이후 [NewLearn Note](https://1dohyeon.github.io/projects/newlearnnote)에 안정적으로 통합하는 전략을 선택했습니다.

## 내가 맡은 것

개인 프로젝트로 진행했으며, **FastAPI 백엔드 전반과 Next.js 프론트엔드**를 담당했습니다. RAG 파이프라인 설계와 구현, 비동기 문서 처리 시스템, GCS 파일 저장, 채팅 인터페이스까지 전 영역을 개발했습니다.

## 개발 스토리

### 1. NestJS 대신 FastAPI를 선택한 이유

메인 프로젝트인 NewLearn Note에서는 NestJS를 사용하고 있어서 Nura도 같은 스택으로 갈 수 있었습니다. 하지만 LangChain, ChromaDB, OpenAI SDK가 모두 Python 기반이라, NestJS에서 이를 연동하려면 별도 Python 프로세스를 띄우거나 외부 API로 감싸야 했습니다. 프로토타입 검증이 목적인데 불필요한 복잡도를 더하고 싶지 않았습니다.

FastAPI를 선택하면 Python AI 생태계를 네이티브로 활용할 수 있고, Pydantic 기반 요청/응답 검증과 Swagger 자동 생성도 지원합니다. 익숙한 스택 대신 목적에 맞는 도구를 선택하는 것이 맞다고 판단했습니다.

### 2. RAG 파이프라인 설계

RAG의 전체 흐름은 크게 두 단계입니다. 문서를 처리해서 벡터로 저장하는 **인덱싱 단계**와, 질문이 들어오면 관련 청크를 검색해 LLM에 전달하는 **검색-생성 단계**입니다.

```
[인덱싱] 문서 업로드 → GCS 저장 → 파싱 → 청킹 → 임베딩 → ChromaDB 저장
[검색-생성] 질문 입력 → 질문 임베딩 → 유사도 검색(Top-K) → 프롬프트 구성 → GPT-4o-mini → 답변
```

각 단계를 `DocumentService`와 `RAGService`로 분리해 책임을 명확히 했고, 임베딩 모델은 싱글톤 패턴으로 초기화 비용을 한 번만 부담하도록 했습니다.

벡터 DB로는 ChromaDB를 선택했습니다. 별도 서버 없이 Python 라이브러리로 동작하고 LangChain과 네이티브로 통합되어 프로토타입에 적합했습니다. HNSW 알고리즘 기반으로 수만 건 문서까지 처리할 수 있어 성능도 충분했습니다.

### 3. 청킹 전략 최적화 — 문장이 잘리는 문제

처음에는 문서를 고정 500자 단위로 잘라 벡터로 저장했습니다. 구현이 단순하고 청크 크기가 균일하다는 장점이 있었지만, 실제 질문을 해보니 문제가 드러났습니다. 논문에서 특정 개념을 설명하는 문단 중간에 청크가 잘리면서 LLM에게 맥락 없는 텍스트 조각이 전달되고, 답변 품질이 떨어졌습니다.

LangChain의 `RecursiveCharacterTextSplitter`로 전환해 문단(`\n\n`) → 줄바꿈(`\n`) → 공백 순서로 자연스럽게 분할하도록 바꿨습니다. chunk_size는 1000자로 늘려 맥락을 충분히 담고, chunk_overlap은 200자로 설정해 청크 경계에서 문맥이 끊기지 않도록 했습니다. 이후 논문 PDF에서 특정 섹션 내용을 질문했을 때 해당 부분을 정확히 찾아 답변하는 것을 확인했습니다.

### 4. 프롬프트 엔지니어링 — 문서 무관 질문에 억지 답변하는 문제

초기 프롬프트는 단순했습니다. `이 문서를 읽고 질문에 답해줘.` 수준이었는데, 문서와 전혀 관련 없는 질문("오늘 날씨는?")을 해도 LLM이 업로드된 문서 내용을 억지로 끼워 맞춰 답변하는 문제가 생겼습니다. 사용자 입장에서는 AI가 뭔가 이상한 말을 하는 것처럼 느껴졌습니다.

프롬프트를 두 가지 케이스로 분기했습니다. 관련 문서 청크가 검색된 경우에는 "문서 내용이 질문과 관련 있다면 우선 참고하고, 어떤 문서를 참고했는지 언급하라"고 지시합니다. 검색된 청크가 없거나 관련도가 낮으면 일반 지식으로 답변하도록 했습니다. 여기에 최근 10개 대화 히스토리를 함께 제공해 맥락 있는 연속 대화가 가능하게 했습니다.

### 5. 비동기 문서 처리 — 30초 대기 문제 해결

100페이지 분량의 논문 PDF를 업로드하면 파싱과 임베딩 생성에 약 30초가 걸렸습니다. 처음에는 업로드 요청이 들어오면 모든 처리가 끝날 때까지 응답을 보내지 않는 동기 방식으로 구현했는데, 사용자가 30초 동안 빈 화면을 보며 기다려야 했습니다.

FastAPI의 `BackgroundTasks`를 활용해 흐름을 바꿨습니다. 파일이 들어오면 문서 메타데이터만 DB에 저장하고 상태를 `processing`으로 설정한 뒤 즉시 응답을 반환합니다. 실제 파싱과 임베딩은 백그라운드 태스크에서 처리하고, 완료되면 상태를 `completed`로 업데이트합니다. 처리 중 실패하면 `failed`로 기록해 재시도가 가능하도록 했습니다. 사용자는 업로드 즉시 다른 작업을 계속할 수 있게 됐습니다.

### 6. GCS 도입 — 서버 재배포마다 문서가 사라지는 문제

초기에는 업로드된 파일을 서버 로컬 파일시스템에 저장했습니다. 개발 중에는 문제가 없었는데, 코드를 수정해서 서버를 재배포하는 순간 모든 문서가 사라졌습니다. 컨테이너 기반 환경에서는 로컬 파일이 영속적으로 유지되지 않는다는 점을 간과한 것이었습니다.

GCS(Google Cloud Storage)로 전환해 파일을 외부 스토리지에 저장하도록 했습니다. 서버 재배포와 무관하게 문서가 보존되고, 서버 스토리지 부담도 줄었습니다. 나중에 NewLearn Note와 통합할 때도 같은 GCS 인프라를 공유할 수 있어 연동이 수월합니다.

## 배운 점

RAG 시스템을 처음 구현하면서 각 단계가 독립적으로 동작하는 것처럼 보여도 실제로는 청킹 전략이 검색 품질을 결정하고, 검색 품질이 LLM 답변 품질을 결정하는 연쇄 구조라는 것을 체감했습니다. 어느 한 단계를 잘못 설계하면 전체 파이프라인이 흔들렸고, 각 단계를 독립 서비스로 분리한 덕분에 문제 지점을 빠르게 찾을 수 있었습니다.

프롬프트 엔지니어링이 생각보다 훨씬 중요하다는 것도 배웠습니다. 모델 자체의 성능보다 어떤 맥락을 주고 어떻게 지시하느냐가 답변 품질에 더 큰 영향을 미쳤습니다. "잘 답해줘"보다 "언제 문서를 쓰고 언제 쓰지 말아야 하는지"를 명확히 지시하는 것이 핵심이었습니다.

익숙한 NestJS 대신 FastAPI를 선택한 것도 좋은 경험이었습니다. 도구를 프로젝트에 맞춰 선택하는 것과, 자신이 편한 도구에 프로젝트를 맞추는 것은 결과물의 완성도에서 차이가 납니다.

향후 Nura에서 검증한 RAG 파이프라인을 [NewLearn Note](https://1dohyeon.github.io/projects/newlearnnote)에 통합하여, Next.js 웹/Electron 데스크톱 앱에서 AI 어시스턴스 UI를 구현할 예정입니다.
